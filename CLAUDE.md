## ファイル編集のルール

### Markdown スタイル
- 見出しは `#` `##` `###` を使い、階層を守る
- テーブルはパイプ記法 `| col | col |` を使う
- 日本語コンテンツが基本。固有名詞（ホテル名・駅名）は英語表記も併記する
- 変更内容は必ず日本語でコミットメッセージを書く

### 日程ファイル（schedule_*.md）
- 1ファイル = 1航空会社プランに対応
- フォーマット変更は全ファイルに統一適用する
- チェックリスト `- [ ]` は予約済みを `- [x]` に変える

### ホテルリスト（hotels.md）
- 1候補 = 1ブロック（ホテル名・立地・価格・予約リンク・コメント）
- 価格は `USD/泊` と `JPY換算` を両方記載する
- 評価スコアは 5段階（★★★★★）

## SDD（スペック駆動開発）ワークフロー

本プロジェクトでは以下の手順で機能追加・大きな変更を行います。

### ステップ 1: Issue で目的を定義
GitHub Issue に「何を・なぜ変えたいか」を書く。
→ ラベル: `spec-needed`

### ステップ 2: Spec ファイルを作成
`specs/NNN-<slug>.md` を作成し、以下を記述:
- 目的・背景
- 受け入れ条件（Acceptance Criteria）
- 変更対象ファイル
- 制約事項

### ステップ 3: Spec レビュー
PR を出してチームメンバーにレビューしてもらう（コードの前に仕様を承認）。
→ ラベル: `spec-review`

### ステップ 4: 実装
Spec が承認されたら実装ブランチを切って作業開始。
→ ラベル: `in-progress`

### ステップ 5: 実装 PR
実装 PR には「Closes #<issue番号>」と「参照 spec: specs/NNN-*.md」を記載。

## AI エージェントへの指示

### Claude Code がやってよいこと
- Markdown ファイルの内容編集・追記
- specs/ ディレクトリへのスペックファイル作成
- リンクの追加・整理
- 日程表のフォーマット統一

### Claude Code がやってはいけないこと
- 予約の確定・キャンセル（外部サービスへの書き込み）
- ホテルや航空券の価格を「確認せず」に変更
- main ブランチへの直接プッシュ
- チームメンバーへの代理メッセージ送信

### よく使うコマンド
```bash
# 全スケジュールファイルのチェックリスト状況を確認
grep -n "\- \[" schedule_*.md

# ホテルリストの価格帯を確認
grep -n "USD" hotels.md

# 未決定事項の一覧
grep -rn "TODO\|TBD\|未定" *.md
```
