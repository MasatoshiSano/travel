# GitHub × CLAUDE.md × SDD チーム開発ベストプラクティス（2026年版）

> このドキュメントは「旅行計画リポジトリ」を具体例として、
> AI エージェント（Claude Code）と人間が協調するチーム開発の手法を解説します。

---

## 1. 全体像：3つの柱

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   GitHub Issues/PR          CLAUDE.md               │
│   ─────────────────         ──────────              │
│   ・目的・議論の場            ・AI への指示書           │
│   ・変更履歴                 ・チームルール集           │
│   ・レビュー                 ・育て続けるドキュメント    │
│            ↘                    ↙                  │
│              ┌──────────────┐                       │
│              │  SDD         │                       │
│              │  スペック駆動 │                       │
│              │  開発        │                       │
│              └──────────────┘                       │
│              コードより先に「仕様」を合意する           │
└─────────────────────────────────────────────────────┘
```

**なぜこの3つが必要か？**

AI エージェントは「指示通りに動く」が「文脈を自動共有する」能力はない。
CLAUDE.md で文脈を渡し、SDD で仕様を事前合意し、GitHub で非同期コラボレーションする。

## 2. CLAUDE.md の育て方

### 2-1. 最小構成から始める（Day 1）

最初は5行でよい。完璧を求めない。

```markdown
# CLAUDE.md

## プロジェクト概要
イタリア・フランス旅行の計画リポジトリ。3名で共同編集。

## やってはいけないこと
- main ブランチへの直接プッシュ
- 外部サービスへの予約・キャンセル操作
```

### 2-2. 「AI が間違えたとき」に育てる（継続）

AI が間違った操作をするたびに CLAUDE.md に追記する。
これが最も効果的な育て方。

**悪い例（ドキュメントだけ直して終わり）:**
```
AI がホテル価格を古い情報で書き換えてしまった
→ 手動で直して終了
```

**良い例（CLAUDE.md に学習させる）:**
```
AI がホテル価格を古い情報で書き換えてしまった
→ 手動で直す
→ CLAUDE.md に追記：
  "ホテルの価格は「確認せず」に変更しない。
   必ず booking.com / 公式サイトへのリンクを確認してから更新すること"
```

### 2-3. セクション構成のベストプラクティス

```markdown
# CLAUDE.md

## プロジェクト概要        ← 必須。AI に文脈を与える
## ディレクトリ構成        ← ファイル追加のたびに更新
## 編集ルール              ← Markdown スタイル・命名規則
## Git ワークフロー        ← ブランチ名・コミットメッセージ規則
## SDD ワークフロー        ← スペック駆動開発の手順
## AI へのルール           ← やってよいこと / やってはいけないこと
## よく使うコマンド        ← grep・git コマンドのスニペット
```

### 2-4. CLAUDE.md の更新タイミング

| タイミング | 何を追記するか |
|----------|-------------|
| 新ファイル追加 | ディレクトリ構成セクション |
| チームルール決定 | 編集ルールセクション |
| AI が誤操作 | 禁止事項セクション |
| 便利コマンド発見 | よく使うコマンドセクション |
| 新メンバー参加 | プロジェクト概要（メンバー一覧） |

## 3. SDD（スペック駆動開発）の実践

### 3-1. SDD とは

「コードを書く前に仕様書（Spec）をレビュー・承認してもらう」開発手法。
AI エージェントを使う場合は特に重要。

```
従来の開発:
Issue → 実装 → レビュー → マージ

SDD:
Issue → Spec作成 → Specレビュー・承認 → 実装 → 実装レビュー → マージ
                    ↑ここで方向性を合意する
```

**なぜ AI 時代に SDD が重要か？**
AI はコードを速く大量に書く。方向性の合意なしに実装すると、
「大量の間違った実装」が生まれてしまう。Spec で先に合意することで無駄を防ぐ。

### 3-2. Spec ファイルの書き方

`specs/NNN-<slug>.md` 形式で管理する。

**必須セクション:**

```markdown
# Spec NNN: <タイトル>

**ステータス:** Draft | In Review | Approved | Implemented
**作成者:** <名前>
**関連 Issue:** #<番号>

## 背景・目的
なぜこの変更が必要か。

## 受け入れ条件（Acceptance Criteria）
- [ ] 条件1
- [ ] 条件2
（すべて満たされたら「完了」）

## 変更対象ファイル
何を変えるか。何を変えないか（スコープ外）も明記。

## 実装メモ（Claude Code 向け）
AI が実装する際の具体的な指示・フォーマット例。
```

### 3-3. Spec のライフサイクル

```
Draft（作成）
  ↓
spec-review（PR で全員レビュー）
  ↓
Approved（全員 approve）
  ↓
in-progress（実装開始）
  ↓
Implemented（実装PR マージ）
```

### 3-4. 具体的なフロー例

**例: 旅行リポジトリでホテルフォーマットを変更したい場合**

```bash
# Step 1: Issue 作成
# GitHub で Issue #12「ホテルリストのフォーマット統一と絞り込み」を作成
# ラベル: spec-needed

# Step 2: Spec ブランチを切る
git checkout -b spec/12-hotel-selection

# Step 3: Spec ファイルを作成
# specs/001-hotel-selection.md を書く（受け入れ条件を明確に）

# Step 4: Spec PR を出す
# PR タイトル: "Spec: ホテル最終選定フォーマット #12"
# PR 本文: "コードは一切含みません。仕様の合意だけをお願いします。"

# Step 5: チームレビュー → Approve → マージ

# Step 6: 実装ブランチを切る
git checkout -b feature/12-hotel-selection

# Step 7: Claude Code に実装させる
# 「specs/001-hotel-selection.md の受け入れ条件に従って hotels.md を更新して」

# Step 8: 実装 PR
# PR 本文に「Closes #12」「参照 spec: specs/001-hotel-selection.md」を記載
```

## 4. GitHub チーム開発のベストプラクティス

### 4-1. ブランチ戦略

```
main（保護ブランチ、直接プッシュ禁止）
  ├── spec/12-hotel-selection    ← Spec PR 用
  ├── feature/12-hotel-selection ← 実装 PR 用
  ├── fix/15-broken-link         ← バグ修正用
  └── claude/xxx                 ← Claude Code が自動作成するブランチ
```

**Branch Protection Rules（必須設定）:**
- `main` ブランチへの直接プッシュを禁止
- PR にレビュー承認を1件以上要求
- 「Require branches to be up to date」を有効化

### 4-2. Issue テンプレート

`.github/ISSUE_TEMPLATE/` に以下を用意する：

**spec-request.md（仕様変更要求）:**
```markdown
## 目的
何を達成したいか

## 背景
なぜ必要か

## 提案内容
どう変えたいか

## 影響範囲
どのファイルに影響するか

## 完了条件
何をもって完了とするか

/cc @Masa @Yuki @Kenta
```

### 4-3. PR テンプレート

`.github/pull_request_template.md`:
```markdown
## 変更内容
（何を変えたか）

## 関連 Issue
Closes #

## 参照 Spec
specs/NNN-*.md

## テスト・確認方法
（どう確認したか）

## レビュアーへのメモ
（特に見てほしい箇所）
```

### 4-4. ラベル設計

| ラベル | 意味 |
|-------|------|
| `spec-needed` | Spec 作成が必要 |
| `spec-review` | Spec のレビュー待ち |
| `spec-approved` | Spec 承認済み |
| `in-progress` | 実装中 |
| `ai-generated` | Claude Code が生成したコード |
| `needs-human-review` | 人間のレビューが特に必要 |
| `claude-ready` | Claude Code に任せてよいタスク |

### 4-5. Claude Code を使ったレビューフロー

```bash
# PR のレビューを Claude Code に依頼する例
# （claude.ai/code または IDE 拡張で）

# 「PR #23 のdiffを見て、
#   specs/001-hotel-selection.md の受け入れ条件をすべて満たしているか確認して」
```

## 5. CLAUDE.md × SDD × GitHub の連携パターン

### パターン A: 小さな変更（ファイル追加・リンク追加）

```
Issue（任意） → feature ブランチ → 実装 → PR → レビュー → マージ
```
Spec は不要。コミットメッセージで変更内容を説明する。

### パターン B: 中規模変更（フォーマット変更・既存ファイルの大幅更新）

```
Issue → Spec 作成 → Spec PR → 承認 → 実装ブランチ → PR → マージ
```

### パターン C: 大規模変更（アーキテクチャ変更・全ファイル影響）

```
Issue → 議論・RFC → Spec 作成 → Spec PR（全員 approve）
→ 実装を複数PRに分割 → 順番にマージ → CLAUDE.md 更新
```

## 6. よくあるアンチパターンと対策

### ❌ CLAUDE.md を更新しない

**問題:** AI が古いルールで動き続ける。チームルールと実際の動作が乖離する。
**対策:** ルール変更のたびに必ず CLAUDE.md を更新するコミットを含める。

### ❌ Spec なしに AI に実装させる

**問題:** AI が「それっぽいが間違った」実装を大量生成する。手戻りが大きい。
**対策:** 「受け入れ条件」を Spec に書いてから AI に実装依頼する。

### ❌ AI の出力をレビューせずにマージする

**問題:** 誤った情報・不整合・想定外の変更がマインブランチに入る。
**対策:** `ai-generated` ラベルを付け、必ず人間がレビューする。

### ❌ CLAUDE.md が肥大化して読まれなくなる

**問題:** 長すぎると AI も人間も読まない。
**対策:**
- 1500行を超えたら分割を検討（`docs/` に詳細を移す）
- CLAUDE.md は「重要なルールのインデックス」にとどめる
- 古くなったルールは削除する（ゾンビルール禁止）
